# RFC 7413 — TCP Fast Open

| Field       | Value                            |
|-------------|----------------------------------|
| RFC number  | 7413                             |
| Title       | TCP Fast Open                    |
| Category    | Experimental                     |
| Date        | December 2014                    |
| Source text | [`rfc7413.txt`](rfc7413.txt)     |

This document records, paragraph by paragraph, how the
current PyTCP codebase relates to each normative
statement in RFC 7413. The audit was performed by
reading the RFC text fresh and inspecting the codebase
under `pytcp/protocols/tcp/`, `pytcp/socket/`,
`pytcp/stack/`, and `net_proto/protocols/tcp/options/`
directly; no prior memory or rule-file content was
reused. Sections that contain no normative content
(Abstract, §1 Introduction narrative, §2 motivation,
§3 protocol overview, §6 applicability, §7 open
experimentation, §8 related work, §9 IANA, §10
references) are omitted.

---

## §4.1.1 Fast Open Option (wire format)

> "Kind: 1 byte, value = 34 / Length: 1 byte, range
> 6 to 18 / Cookie: 0, or 4 to 16 bytes"

**Adherence:** met. The wire-level encoding at
`net_proto/protocols/tcp/options/tcp__option__fastopen.py`
implements kind=34 and the variable-length cookie
field (4-16 bytes) per the RFC.

> "Options with invalid Length values or without the
> SYN flag set MUST be ignored."

**Adherence:** met. The TFO option emission at
`pytcp/protocols/tcp/tcp__session.py:1465-1474` only
produces the option on SYN-bearing segments (the
gates `flag_syn and flag_ack` for SYN+ACK and
`flag_syn and not flag_ack` for active-open SYN).
Non-SYN segments never emit the option.

The "ignore on receipt" rule is implemented
implicitly: the session only consults the inbound
TFO option in the SYN-handling FSM states, never on
established-state segments.

## §4.1.2 Server Cookie Handling

### Cookie generation properties

> "1. The cookie authenticates the client's (source)
>     IP address of the SYN packet.
>  2. The cookie can only be generated by the server
>     and cannot be fabricated by any other parties.
>  3. The generation and verification are fast..."

**Adherence:** met. PyTCP's
`generate_cookie` at
`pytcp/protocols/tcp/tcp__fastopen.py:46-66`:

```python
digest = hmac.new(secret, bytes(peer_address), "sha256").digest()
return digest[:TCP__FASTOPEN__COOKIE_LEN]
```

- Property 1: cookie is bound to peer IP (passed as
  `peer_address`), satisfying "authenticates client
  IP".
- Property 2: HMAC-SHA256 with a server-secret key
  (16-byte `TCP__FASTOPEN_SECRET` from
  `pytcp/stack/__init__.py:91`) — unforgeable
  without the secret.
- Property 3: HMAC-SHA256 is fast (microseconds on
  modern CPUs).

### Cookie length

> "The minimum cookie size is 4 bytes."

**Adherence:** met. `TCP__FASTOPEN__COOKIE_LEN = 8`
(line 41) — within the 4-16 byte range.

### Cookie expiration

> "5. The cookie expires after a certain amount of
>     time."

**Adherence:** met implicitly by secret rotation.
PyTCP's `TCP__FASTOPEN_SECRET` is generated at
process startup; restarting the process invalidates
all outstanding cookies. Within a process the secret
is stable. The §4.1.2 clause is permissive ("The
reason for cookie expiration is detailed... This
can be done by either periodically changing the
server key... or including a timestamp"); PyTCP
uses the per-process secret approach.

### Cookie validation

> "IsCookieValid(IP_Address, Cookie): checks if the
> cookie is valid..."

**Adherence:** met. `validate_cookie` at
`pytcp/protocols/tcp/tcp__fastopen.py:69-79` uses
`hmac.compare_digest` for constant-time comparison
to prevent timing side-channels.

## §4.1.3 Client Cookie Handling

### Client MUST cache cookies

> "The client MUST cache cookies from servers for
> later Fast Open connections."

**Adherence:** met. The cache is at
`stack.tcp__fastopen_cookies` (a dict mapping
`peer_ip -> cookie`); the client populates it on
receiving a SYN+ACK with a TFO cookie.

### Per-IP caching, not per-port

> "The client should cache at most one (most recently
> received) cookie per client and server IP address
> pair."

**Adherence:** met. The cache is keyed by
`peer_ip_address` only, not (ip, port). This matches
the §4.1.3 "per IP address pair" guidance.

### Cache MSS too (recommendation)

> "We recommend that the client also cache the
> Maximum Segment Size (MSS) advertised by the
> server."

**Adherence:** n/a (non-normative). The recommendation
is "we recommend" not MUST/SHOULD, so the omission is
permissible by the RFC's own framing. Practical impact:
the first TFO connection's data length on SYN may
exceed the actual server MSS, requiring
retransmission; subsequent post-handshake
transmissions use the actual negotiated MSS.

---

## §4.1.3.1 Client Caching Negative Responses

> "The client MUST cache negative responses from the
> server in order to avoid potential connection
> failures."

**Adherence:** met. PyTCP maintains
`stack.tcp__fastopen_negative` (a `set[Ip4Address |
Ip6Address]`) that records peers we have seen TFO
fail with. The `_retransmit_packet_timeout` SYN-RTO
path adds the peer to this set whenever an active-
open TFO SYN times out, and `_transmit_packet`'s
TFO-emit branch checks the set on every active-open
SYN and skips the option entirely for known-bad
peers, so the next connection to that peer falls
through to plain 3WHS.

## §4.2 Fast Open Protocol

### Server: FastOpenEnabled flag

> "FastOpenEnabled: default is off. It MUST be turned
> on explicitly by the application."

**Adherence:** met. `_tcp_fastopen_qlen: int = 0`
(line 153) — defaults to 0 meaning "TFO disabled".
Set via `setsockopt(IPPROTO_TCP, TCP_FASTOPEN, qlen)`
to a positive value to enable. The MUST default-off
is satisfied.

### Server: PendingFastOpenRequests limit

> "PendingFastOpenRequests: tracks the number of TFO
> connections in SYN-RCVD state. If this variable
> goes over a preset system limit, the server MUST
> disable TFO for all new connection requests..."

**Adherence:** met. The listen handler in
`tcp__fsm__listen.py` checks
`stack.tcp__fastopen_pending_count <
listen_socket._tcp_fastopen_qlen` before accepting
TFO. The counter is incremented on TFO acceptance
and decremented in `_change_state` when the session
leaves SYN_RCVD (either to ESTABLISHED or to
CLOSED), tracked via the per-session
`_fastopen_pending_counted` guard. Over-limit
incoming SYNs fall back to plain 3WHS.

### §4.2.1 Cookie request flow

> "Step 1: client sends SYN with TFO option, empty
> cookie. Step 2: server responds with SYN-ACK
> carrying TFO option with cookie."

**Adherence:** met. The client-side empty-cookie
request is at `tcp__session.py:1474` (cache lookup
returns empty bytes when no entry); the server-side
cookie generation is at the FSM listen handler
which calls `generate_cookie` and stashes the
result in `_fastopen_cookie_to_emit`.

### §4.2.2 TCP Fast Open data path

> "Client: Send a SYN packet... Include the Fast Open
> option with the cookie of the server. Include any
> data up to the cached server MSS or default 536
> bytes."

**Adherence:** met. The client connect() at
`pytcp/socket/tcp__socket.py:464-470` accepts a
`data` argument; when `data` is non-empty AND a
cookie is cached, the SYN carries both the cookie
and the data. The pre-load goes into
`_tx_buffer` before the FSM transitions to SYN_SENT.

### Server: validate cookie + accept SYN-data

> "Server: Receiving SYN and responding with SYN-ACK"

**Adherence:** met. The FSM listen handler at
`tcp__fsm__listen.py` validates an inbound TFO
cookie via `validate_cookie`; if valid, the server
accepts the SYN-data as if the connection were
established (per §4.2.2 step 4).

---

## §4.4 SYN Retransmit Without TFO

> "When the SYN timer fires, the client SHOULD
> retransmit a SYN packet without data and Fast Open
> options."

**Adherence:** met. The `_retransmit_packet_timeout`
SYN-RTO path sets `_fastopen_syn_retransmitted = True`
on the active-open session. `_transmit_packet`'s
active-open SYN branch reads the flag and suppresses
the TFO option emission, so the SYN retransmit goes
out as plain 3WHS without the TFO option or any
SYN-piggybacked data.

PyTCP implements the core RFC 7413 TFO mechanism
including bilateral negotiation, cookie generation
and validation, server-side cookie issuance,
client-side cookie caching, and the data-on-SYN
fast path. Three substantive gaps:

1. **§4.1.3.1 negative response caching (MUST)**:
   not implemented. Repeated TFO failures to the
   same server are not cached; the client retries
   TFO indefinitely.

2. **§4.2 PendingFastOpenRequests limit (MUST)**:
   the application-supplied `qlen` is stored but
   not actively enforced as a SYN-RCVD count.
   PyTCP can be DoS'd via TFO-cookie-validation
   resource exhaustion under sufficient attacker
   load.

3. **§4.4 SYN retransmit without TFO (SHOULD)**:
   PyTCP does not strip TFO from retransmitted SYNs.

All three gaps are localised fixes (~5-10 LOC each
plus tests). The §4.2 limit gap is the most
security-relevant; closing it would require
incrementing/decrementing a counter on every SYN-
RCVD entry/exit and rejecting new TFO attempts
when the counter exceeds the configured `qlen`.
